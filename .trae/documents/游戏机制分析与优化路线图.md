# Evertrail 游戏机制分析与优化路线图

## 1. 摘要（Summary）

Evertrail 当前已从“背景地图 + 节点”演进为一个可玩的 2D 像素风平台探索世界：

- 玩家通过 `A/D` 移动、`Space/W` 跳跃、`E/Enter` 与地图元素交互。
- 世界基于日记条目（mood + tags）程序化生成，包含地形、生态、路径点、收集物、章节门。
- 渲染管线按层绘制：视差背景 → 背景装饰 → 前景方块 → 实体 → 粒子 → 光照/后期 → UI。

本计划先梳理当前机制与生成逻辑，再按 **性能与渲染、玩法与内容、世界生成与生态、体验与可访问性** 四个方向给出可在后续对话中分别执行的详细提示词（prompt）。

### 2.2 当前游戏机制

#### 移动与物理
- 横向加速 + 速度上限 + 地面/空中摩擦力（[Player.ts L48-L82](file:///d:/Evertrail/src/lib/game/Player.ts#L48-L82)）
- 仅在地面时可跳跃（`onGround` 探测）
- 重力与最大下落速度限制
- AABB 碰撞：水平/垂直各 3 点采样，遇到方块即贴边（[Player.ts L133-L198](file:///d:/Evertrail/src/lib/game/Player.ts#L133-L198)）

#### 相机
- 默认跟随玩家，`x += (targetX - x) * 0.08` 平滑插值
- 鼠标拖拽进入 `free` 模式；玩家再次移动后恢复自动跟随（[MapCanvas.tsx L313-L321](file:///d:/Evertrail/src/components/MapCanvas.tsx#L313-L321)）
- 已加入 `Number.isFinite` 防御非有限坐标

#### 交互
- `checkInteraction` 检测最近的交互对象：收集物 / 路径点 / 章节门，半径分别为 `40/40/80`（[InteractiveObjects.ts L295-L331](file:///d:/Evertrail/src/lib/game/InteractiveObjects.ts#L295-L331)）
- 按下 `E/Enter` 触发：收集物产生 sparkle 粒子并播放音效；路径点打开对应日记条目并激活；章节门进入章节回忆
- 鼠标点击路径点（半径 40）也能打开回忆

#### 地图元素
- **路径点（Waypoint）**：最后一个默认已激活（篝火），其余为石碑；稀有度 ≥3 带光环（[InteractiveObjects.ts L42-L62](file:///d:/Evertrail/src/lib/game/InteractiveObjects.ts#L42-L62)）
- **收集物（Collectible）**：根据 entry.tags 决定种类（book/footprint/heart/star/leaf/flame/gem/camera），每个 entry 最多 3 个，悬浮动画
- **章节门（ChapterGate）**：横跨 chapter 起止节点，带主题色与标题
- **树实体（TreeEntity）**：在 grass 方块上按 12% 概率生成， sway 动画

### 2.3 世界是如何生成的

1. **节点生成**：`generateMapNodes` 按 `entries` 顺序排列，间距 `160`，x 带 jitter，y 由 Simplex Noise 地表高度决定，biome 取自 `entry.mood`（[mapGenerator.ts L15-L32](file:///d:/Evertrail/src/lib/mapGenerator.ts#L15-L32)）。
2. **地形生成**：`generateWorld` 在 `[minX, maxX]` 范围内逐 `TILE_SIZE` 生成：
   - 用 `noise2D(wx * 0.005, 0)` 计算地表 Y（`WORLD_GROUND_Y + round(noise*60/32)*32`）
   - 每个 x 列生成 1 层地表 + 8 层地下
   - 根据最近节点 mood 决定 `biome` 与地表类型（grass/mud/ash/sand/mycelium）
   - 在节点正下方生成非实体的 `platform` 石碑基座（[World.ts L16-L75](file:///d:/Evertrail/src/lib/game/World.ts#L16-L75)）
3. **生态调色**：`BIOME_PALETTES` 定义了天空、地表、树木、点缀色、环境粒子类型（[BiomeRenderer.ts L4-L89](file:///d:/Evertrail/src/lib/game/BiomeRenderer.ts#L4-L89)）。
4. **实体验证**：`buildWaypoints/buildCollectibles/buildChapterGates/buildTreeEntities` 把节点/条目/章节转换成可渲染实体（[InteractiveObjects.ts](file:///d:/Evertrail/src/lib/game/InteractiveObjects.ts)、[Entity.ts L340-L352](file:///d:/Evertrail/src/lib/game/Entity.ts#L340-L352)）。

### 2.4 渲染流程

每一帧在 [RenderPipeline.render](file:///d:/Evertrail/src/lib/game/RenderPipeline.ts#L69-L110) 中执行：

1. `clearRect`
2. 视差背景（`ParallaxBackground.render`）
3. 背景装饰（`TileRenderer.drawDecorations`）
4. 液体层（预留）
5. 前景方块（`TileRenderer.draw`）
6. 实体/NPC（`EntityRenderer.render`）
7. 粒子（`ParticleSystem.render`）
8. 光照叠加（`LightingSystem.apply`，multiply 混合）
9. 后期处理（`PostProcessSystem.render`）
10. UI（`MapCanvas` 传入的 `drawUI`）

### 方向 1：性能与渲染优化

#### 目标
让大地图（更多 entries / 更长世界）在 60fps 下稳定运行，减少每帧 CPU/GPU 重复工作。

#### 涉及文件
- [World.ts](file:///d:/Evertrail/src/lib/game/World.ts)
- [TileRenderer.ts](file:///d:/Evertrail/src/lib/game/TileRenderer.ts)
- [LightingSystem.ts](file:///d:/Evertrail/src/lib/game/LightingSystem.ts)
- [EntityRenderer.ts](file:///d:/Evertrail/src/lib/game/EntityRenderer.ts)
- [RenderPipeline.ts](file:///d:/Evertrail/src/lib/game/RenderPipeline.ts)

#### 详细提示词

```text
在 d:\Evertrail 项目中执行性能与渲染优化，要求如下：

1. 空间索引
   - 修改 src/lib/game/World.ts，为 WorldData 增加一个基于 TILE_SIZE 网格的 Map<string, Tile>（key 为 `${tx},${ty}`）。
   - 在 generateWorld 生成 tiles 后填充该索引。
   - 重写 tileAt / isSolidAt / getTilesInRange，优先使用索引查询，保证复杂度从 O(n) 降到近似 O(1)。
   - 注意保持原有接口签名不变，避免影响 Player.ts 与 TileRenderer.ts。

2. 方块渲染缓存
   - 修改 src/lib/game/TileRenderer.ts：
     a. 把 drawTile 中“每个方块都 createLinearGradient”改为使用一个预计算的 Map<string, CanvasGradient>，key 包含 tile.type + biome + sx/sy 的相对位置或颜色，避免每帧创建渐变。
     b. 把装饰物（drawDecorations）的随机决策从“每帧重新随机”改为在 generateWorld 阶段就把装饰类型写入 tile 的 metadata（如 tile.decoration），或单独生成 Decoration[] 列表；TileRenderer 只负责绘制，不再做随机。
   - 可选：为常见方块类型生成离屏精灵 canvas（32x32 或 64x64），drawTile 时优先 drawImage；仅在相机缩放或调色板变化时重绘。

3. 光照裁剪
   - 修改 src/lib/game/LightingSystem.ts：
     a. 在 apply 中加入光源视口裁剪，只处理 x 在 [-radius, viewWidth+radius]、y 在 [-radius, viewHeight+radius] 的光源。
     b. 对 moon 光源这种超大半径，改为用全屏渐变或环境色直接叠加，不再走径向渐变逐像素。
     c. 确保 Number.isFinite 防御保留。

4. 渲染管线批量
   - 修改 src/lib/game/EntityRenderer.ts：
     a. 在 render 中先收集可见实体，再按是否使用 shadowBlur 分组批量绘制，减少 state change。
     b. 合并同类简单实体（如 CollectibleEntity）可考虑统一绘制函数，但保留 Entity 抽象。

5. 视口外剔除
   - 检查 RenderPipeline 中各层是否都使用 RENDER_MARGIN 进行有效裁剪，补齐缺失的剔除。

6. 验证
   - 运行 npm run check && npm run lint && npm run build，必须全部通过。
   - 在浏览器 DevTools Performance 中录制 10 秒移动/跳跃，确保每帧主线程时间 < 16ms，无明显掉帧。
   - 保持现有视觉风格不变，不破坏 autotile、粒子、光照效果。
```

#### 验收标准
- `npm run check/lint/build` 通过。
- 100 条日记条目生成地图后，移动端与桌面端均保持 60fps。
- 现有视觉无退化。

### 方向 3：世界生成与生态

#### 目标
让地形更丰富（洞穴、水体、平台、生态过渡），从“一条地表走廊”变成有纵深的 Terraria-like 世界。

#### 详细提示词

```text

1. 地形分层与洞穴
   - 修改 src/lib/game/World.ts 的 generateWorld：
     a. 在地下 3-5 层使用第二个 Simplex Noise（不同 seed/频率）挖空洞穴，形成连通的地下空腔； cave 区域内的 solid 改为 false。
     b. 在洞穴底部和地表之间生成小平台（platform tile），让玩家可以跳到高处或下到地下探索。
     c. 保证玩家出生点（最新节点）周围 3 格内不会被挖空，避免坠落即死。

2. 水体与液体层
   - 在 src/lib/game/RenderPipeline.ts 中取消“液体层（预留）”注释，实际调用新的 renderLiquids 方法。

   - TileRenderer 在生态边界处按混合调色板绘制地表顶层。

4. 植被群落与装饰密度
   - 在 generateWorld 阶段就根据 biome 与随机种子决定每个 grass/sand/ash/mycelium tile 是否生树/花/仙人掌/蘑菇，并写入 tile.decoration；TileRenderer.drawDecorations 只读取，不再随机。
   - 让 joy/calm 区域树木更密集，tired 区域更稀疏，angry 区域是枯树/岩石。

5. 垂直玩法
   - 扩展相机边界：允许相机向下移动到地下 800px，向上到天空 -400px。
   - 在地下放置额外收集物（稀有 gems），鼓励探索。

6. 验证
   - npm run check/lint/build 通过。
   - 玩家仍可在所有地形正常移动、跳跃、交互；不会因洞穴生成被卡住或无限坠落。
   - 视觉过渡自然，无明显色块突变。
```

#### 验收标准
- 世界生成可配置（洞穴密度、水位高度等参数化）。

## 4. 假设与决策（Assumptions & Decisions）

1. **实现顺序**：四个方向互相独立，但建议先做 **方向 1 性能优化**，再做 **方向 3 世界生成**（洞穴/水体会增加碰撞复杂度，依赖空间索引）。
2. **不破坏现有控制**：所有改动必须保留 `A/D` 移动、`Space/W` 跳跃、`E/Enter` 交互（项目 hard constraint）。
3. **React 与 Canvas 分离**：高频游戏状态继续使用 ref，避免重渲染；设置面板等 UI 用 React 组件。
4. **依赖可用**：允许使用已安装的 `simplex-noise`、`localforage`、`lucide-react`、`zustand`；不引入新的重型游戏引擎。
5. **视觉风格保持像素/Terraria-like**：不转向 3D 或矢量扁平风。
6. **后续执行方式**：每个方向可在单独对话中交给子代理/助手，直接复制对应“详细提示词”。

## 5. 验证步骤（Verification）

无论执行哪个方向，都需完成：

1. `npm run check`（TypeScript 无错）
2. `npm run lint`（ESLint 无错）
3. `npm run build`（生产构建成功）
4. 在浏览器中手动验证：
   - 玩家移动、跳跃、交互正常
   - 相机跟随/拖拽/恢复正常
   - 路径点、收集物、章节门仍可触发
   - 新增功能按预期工作
5. 若涉及性能优化：使用 DevTools Performance 录制 10 秒，目标帧时间 < 16ms。
